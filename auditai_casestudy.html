01

The problem was not model accuracy.
It was the explanation layer.

While working with financial services clients at Deloitte, including Citi, Wells Fargo, and UBS, I repeatedly saw the same gap. Model Risk Managers could validate that an AI model worked technically, but struggled to produce documentation that compliance teams and regulators could actually use.

The issue was not that the models were wrong. It was that the explanation layer between a model output and a regulator question did not exist in a usable form.

The product thesis: compliance teams need regulator-ready audit trails that translate model behavior into plain English, without requiring an ML engineer every time.

02

Three users in the chain.
One primary wedge.

I mapped the model risk review chain and deliberately chose one primary user instead of designing for everyone at once.

Model Risk Manager
Validates the model and produces documentation. Primary user because they are first in the sequence and most frequently blocked.
Compliance Officer + Auditor
Reviews documentation against regulatory standards and verifies decisions after the fact. Both depend on clear audit evidence.

If the Model Risk Manager cannot produce usable documentation, nothing downstream happens: no CCO sign-off, no clean audit trail, and no defensible regulatory position.

03

What I built.
A working audit trail generator.

AuditAI takes a fraud detection model output and generates a structured, plain-English audit trail mapped directly to SR 26-2 documentation requirements.

I trained a Random Forest classifier on the Kaggle Credit Card Fraud Detection dataset and used SHAP to extract feature-level explanations for individual predictions.

The numbers are intentionally not inflated. Of 98 actual fraud cases in the test set, the model caught 81 and missed 17, with 18 false alarms on legitimate transactions. That gap is exactly why the product needs to exist.

04

Three scenarios,
including the uncomfortable one.

Correct fraud catch
Flagged as fraud with 99.96% confidence. Confirmed correct. Top SHAP drivers included V14, V12, V10, V17, and V4.
False positive
Flagged as fraud at 80.8% confidence, but the known outcome was legitimate. AuditAI flags human oversight and risk limitations as failed checks.
Confident legitimate
Correctly classified as legitimate with 0% fraud probability, used as a baseline contrast case.
Why it matters
A tool that only shows correct decisions teaches nothing about explainability. The false positive is the point.
05

Architecture evolved
from deterministic to LLM-assisted.

Fraud model output + SHAP values
↓
AuditAI compliance layer
↓
Claude API generates SR 26-2 structured audit trail as JSON
↓
UI renders checklist, narrative, limitations, and exportable evidence

The first build used deterministic JavaScript templates to assemble audit-trail text from real model outputs. The second build replaced that static template with a live Claude call, passing model metrics and SHAP values to generate the structured compliance narrative.

Being precise about which version does what became part of the product discipline. Explainability tools lose credibility if they overclaim their own reasoning.

06

What I learned.
Trust is a product requirement.

1
MVP sequencing mattered more than feature ambition.
Version history, cross-model dashboards, and regulator Q&A were tempting, but each requires usage history first. Cutting them was sequencing, not shrinking.
2
The North Star had to be compliance, not speed.
Minutes saved is useful, but regulatory completeness is the reason the user has the job. The metric became audit reports meeting SR 26-2 requirements without engineer involvement.
3
A tool that hides failure cases is not trustworthy.
The false-positive scenario is deliberately in the demo because model risk work is about documenting what goes wrong, not only celebrating what goes right.
4
State consistency is a model risk issue.
A mismatch between analysis confidence and exported report confidence became a direct lesson: a beautifully written audit trail is worthless if the number next to it is wrong.
07

What is next.

The next step is to get the prototype in front of someone with real model risk or compliance experience and pressure-test whether the SR 26-2 mapping holds up to scrutiny.

The build path after that is clear: replace the upload-path approximation with a real model inference call, then add V2 features once there is enough usage to justify them: audit trail version history, a cross-model compliance dashboard, and a regulator Q&A simulation mode.
